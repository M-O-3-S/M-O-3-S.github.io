## 📌 주제: AI News Archive 시스템 설계안 3가지 제안 및 최종안 선정

마이클(SW Architect)의 주도로 개발팀이 모여 3가지 아키텍처 대안을 도출하고 장단점을 토론했습니다.

### [제 1안] 순수 로컬 + 정적 호스팅 모놀리스 (선정안)
*   **구조:** Python 스크립트 기반 동작 ➡️ SQLite 저장 ➡️ 로컬 Ollama(gemma3:4B) 추론 ➡️ 생성된 정적 JSON 파일을 GitHub Pages로 Git Push.
*   **스케줄러:** Mac 내장 `cron` (별도 데몬 프로세스 불필요).
*   **👍 장점:** 완전 무료(Zero-Cost). 서버나 DB 클라우드 유지비 없음. 프론트/백 사이의 커플링이 없어 유지보수가 편함.
*   **👎 단점:** 매일 아침 특정 시간에 M4 Mac Mini가 반드시 켜져 있어야 함(절전 모드 해제 필요).

### [제 2안] n8n 워크플로우 + 클라우드 호스팅 조합
*   **구조:** n8n 도커 컨테이너를 상시 띄워두어 자동화 워크플로우를 그리고, DB는 클라우드(Supabase 프리티어)에 적재. 프론트엔드는 Vercel 등에 올려 API로 DB를 찌르는 동적 방식.
*   **👍 장점:** 실행 로깅과 워크플로우의 시각화가 탁월하여 에러 발생 시 UI에서 바로 복구(Retry) 가능. 추후 다른 API(슬랙 등) 확장이 매우 쉬움.
*   **👎 단점:** n8n 컨테이너가 램(RAM) 병목을 유발해 Ollama에 할당할 16GB VRAM을 갉아먹음. 정적 호스팅이 아니므로 프리티어 초과 시 서버 유지비용(DB, Vercel) 발생 위험성.

### [제 3안] 완전 관리형 Serverless + 외부 상용 AI API (OpenAI)
*   **구조:** AWS Lambda / GCP Cloud Functions에서 파이썬 코드 구동. DB는 DynamoDB 사용. AI는 로컬 Ollama를 버리고 빠르고 성능 좋은 OpenAI (ChatGPT) 외부 API 호출.
*   **👍 장점:** Mac Mini가 꺼져 있어도 클라우드에서 자율적으로 24시간 도는 완벽한 무중단 아키텍처. 퀄리티 높은 요약 속도 및 파싱 에러 제로.
*   **👎 단점:** 극악의 유지 비용. 매일 돌아가는 API 과금 발생, 클라우드 DB 유지 보수 비용 등 "0원 예산 원칙"을 정면으로 위반함.

## 🧠 에밀리 (AI Engineer)의 하드웨어 리소스 검토 (Feasibility Review)

**질문:** "M4 Mac Mini (16GB RAM) 하드웨어 환경에서 `gemma3:4B` 로컬 모델로 시스템을 만족스럽게 운영할 수 있는가?"

**에밀리의 답변: "충분히 가능하며, 아키텍처 구조 덕분에 만족할 만한 퀄리티가 나옵니다."**

1.  **모델 파라미터와 메모리(VRAM) 대응:** 
    *   `gemma3:4B` 모델은 파라미터 수가 약 40억 개 수준으로, Q4(4-bit) 양자화 모델을 Ollama에 얹을 경우 **VRAM을 약 3~4GB만 점유**합니다. 
    *   Mac의 Unified Memory 16GB 환경에서 3~4GB는 아주 여유로운 수치이며, OS와 파이썬 스크립트가 돌아가는 기본 메모리(약 4~6GB)를 제외하더라도 넉넉히 남습니다. 메모리 오버로드로 인한 커널 패닉이나 버벅임은 없습니다.
2.  **추론 속도 (Inference Speed):** 
    *   방금 진행한 PoC 결과에서도 확인했듯 **단일 기사당 4.18초** 만에 결과를 뱉어냈습니다. M4 칩셋의 강력한 NPU(Neural Engine) 덕분에 4B 수준의 모델은 텍스트 요약에 있어 초당 생성 속도(Tokens per second)가 아주 뛰어납니다. (매일 10~20개 기사를 요약하더라도 총 1~2분이면 파이프라인이 소화됩니다.)
3.  **한국어 및 추론 퀄리티:**
    *   `gemma3` 라인업은 구글의 최신 웨이트 조정 덕분에 4B 체급임에도 불구하고 한글 지시(Instruction) 이해도와 요약 능력이 타사 7B~8B(Llama3 등) 못지않습니다.
    *   단지 가끔 마크다운 코드 블록(` ```json `)을 내뱉는 포맷 불안정성이 있지만, 이건 지능형 프롬프트와 백엔드의 "문자열 정규식 삭제(Cleansing)" 필터 콤보로 막았기 때문에(아키텍처 1안의 방어 로직) **최종 사용자가 보는 데이터 퀄리티는 100% 일정**하게 유지될 수 있습니다!

## 🙋‍♀️ 추가 Q&A (LLM 대안 및 텔레그램 연동 방안)

### Q1. (To 에밀리) 16GB Mac Mini 환경에서 `gemma3:4B`보다 품질이 더 좋은 모델 대안이 있는가?
**에밀리 (AI Engineer):** "네, 웹 리서치 결과 16GB 통합 메모리(Unified Memory) 환경에서는 OS 점유분을 뺀 약 10~12GB를 LLM VRAM으로 돌릴 수 있습니다. 따라서 소비자의 퀄리티 눈높이를 높이기 위해 **7B ~ 12B 파라미터 체급의 양자화(Quantized) 모델**로 업그레이드가 충분히 가능합니다.
*   **[대안 1] Llama 3.2 8B (또는 11B):** 요약 및 추론의 교과서적인 모델입니다. Q4_K_M 양자화를 거치면 약 5~6GB의 메모리만 차지하므로 16GB 환경에서 아주 쾌적하게 돌아갑니다.
*   **[대안 2] Gemma 3 9B (또는 12B):** 현재 4B 모델의 형님 격입니다. 12B 모델이라도 Q4 수준 양자화를 먹이면 16GB 램에 쏙 들어갑니다. 한국어 인스트럭션과 포맷팅(JSON) 규수 능력이 4B보다 압도적으로 좋습니다.
*   **[대안 3] Qwen 3 14B / Phi-4 Mini:** 압축된 지식과 추론 능력(STEM)이 매우 뛰어나 요약 품질에 밀도가 있습니다.

**에밀리의 최종 제안:** 우선 `gemma3:4B` 포맷으로 속도(뼈대)를 먼저 검증하고, 시스템이 안정화되면 명령어 한 줄(`ollama run gemma3:9b` 등)만 쳐서 **Llama 3 8B 나 Gemma 3 9B 급으로 모델을 교체(Scale-up)**하는 것을 강력히 추천합니다!"

### Q2. (To 개발팀) 제 1안(Python+cron) 환경에서 텔레그램 알림은 어떻게 구현되는가?
**엠마 (Backend) & 마이클 (Architect):** "n8n과 같은 시각적 자동화 툴이 없더라도 텔레그램 연동은 아주 쉽고 가볍게 구현됩니다!
*   **구현 방법:** Python 내장 모듈인 `requests` 를 사용하여, Telegram Bot API 엔드포인트(`https://api.telegram.org/bot<TOKEN>/sendMessage`)로 POST 요청을 쏘는 코드를 단 10줄 내외로 작성할 수 있습니다.
*   **파이프라인 위치:** 1안의 아키텍처 상, 스크립트는 `수집(crawler)` ➡️ `요약(processor)` ➡️ `파일생성(publisher)` 순서로 도는데, 이 제일 마지막 `publisher.py`가 JSON 파일을 로컬 디렉토리에 찍어냄과 동시에 **완료되었다는 신호와 요약본 3개를 담아 텔레그램 API로 HTTP Request**를 보냅니다.
*   **보안:** 텔레그램 봇 토큰은 절대 코드에 하드코딩하지 않고 `.env` 파일에 숨겨 마이클이 설계한 6단계(보안) 원칙을 확실히 준수합니다."
