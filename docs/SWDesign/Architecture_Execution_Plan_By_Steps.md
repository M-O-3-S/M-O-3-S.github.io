# AI News Archive: 아키텍트의 8단계 실행 계획서 (Execution Plan)

본 문서는 전통적인 소프트웨어 공학의 8단계 아키텍처 설계 방법론에 따라, 앞으로 마이클(SW Architect)과 다른 엔지니어들이 순차적으로 밟아나갈 구체적인 작업(Task)들과 그 난이도를 정의한 로드맵입니다.
사라님(PM)은 각 단계의 난이도를 참고하여, 해당 단계에 적합한 추론 능력(IQ)을 가진 LLM을 적재적소에 배분하여 작업을 지시하시면 됩니다.

---

### 1단계 · 과제 착수 (Initiation)
*   **작업 내용:** 이미 작성된 PRD와 UX 가이드라인, 페르소나 문서들을 리뷰하여 시스템의 제약 조건(예산 0원, Local Mac Mini)과 비즈니스 목표를 최종 식별합니다. 기존 레거시가 없는 신규 과제임을 명확히 합니다.
*   **예상 산출물:** 시스템 한계(Constraints) 및 핵심 목표 요약서.
*   **난이도: 하 (Low)**
    *   *이유:* 주어진 문서를 읽고 요약/분류하는 기본 독해 능력이면 충분합니다. 가벼운 LLM(예: 8B~14B 급)으로도 충분히 수행 가능합니다.
*   **진행 상태:** ✅ 완료 (앞선 대화들로 대체됨)

---

### 2단계 · 요구사항 분석 (Requirements Analysis)
*   **작업 내용:** PRD에 명시된 기능(FR) 외에 비기능 요구사항(NFR: 속도, 무중단, 스토리지 용량)을 수집합니다. 특히 제한된 디스크 용량에서의 SQLite 관리, Local LLM 대기 시간(Timeout) 등의 트레이드오프를 식별합니다.
*   **예상 산출물:** 비기능 요구사항(NFR) 및 품질 속성(Quality Attributes) 명세서.
*   **난이도: 중 (Medium)**
    *   *이유:* 기획서에 명시되지 않은 숨겨진 에러 케이스(Edge Cases)나 인프라적 제약을 추론(Reasoning)해야 하므로 중간 수준의 모델이 필요합니다.
*   **진행 상태:** 🔄 진행 대기

---

### 3단계 · 아키텍처 설계 (Architecture Design)
*   **작업 내용:** Python 모듈(crawler, processor, publisher)의 폴더 트리 및 파일 구조를 엄격히 분할(Decoupling)합니다. 백엔드와 프론트엔드가 데이터를 주고받을 JSON의 포맷(Data Contract) 구조를 설계합니다.
*   **예상 산출물:** 시스템 구조도(Folder Tree), 컴포넌트 간 통신 규약(JSON Schema).
*   **난이도: 상 (High)**
    *   *이유:* 과제의 전체 뼈대를 세우며 파일 간 의존성을 결정하는 가장 중요한 단계입니다. 코드 분할 능력이 뛰어난 최고 성능의 모델(예: Claude 3.5 Sonnet, GPT-4o, Gemini 1.5 Pro) 선택이 필수적입니다.
*   **진행 상태:** 🔄 진행 대기

---

### 4단계 · 아키텍처 검증 (Architecture Evaluation)
*   **작업 내용:** 3단계에서 설계한 구조 중 핵심 리스크(핵심 기술)가 구현 가능한지 제일 얇은 단위로 찔러봅니다(PoC/Spike). 예를 들어, Ollama API가 Local에서 Python으로 잘 호출되는지 더미 프롬프트로 테스트 스크립트를 짜서 실행해 봅니다.
*   **예상 산출물:** 핵심 리스크(Ollama 연동)를 검증한 짧은 PoC 파이썬 스크립트 및 실행 결과.
*   **비고:** `docs/SWDesign/Architecture_Step4_Evaluation_PoC.md` 및 `tests/poc_ollama_call.py` 스크립트 실행을 통한 4개 리스크 조기 검증.
*   **난이도: 상 (High)**
    *   *이유:* 단순 코딩을 넘어 실제 로컬 환경(Ollama REST API)과의 네트워킹 및 비동기 처리를 테스트해야 하므로 코딩/디버깅 역량이 뛰어난 하이엔드 모델이 필요합니다.
*   **진행 상태:** ✅ 완료

---

### 5단계 · 개발 지원 (Development Support)
*   **작업 내용:** 백엔드(엠마)와 AI(벤)에게 아키텍처 가이드라인에 맞춰 본격적인 코딩을 지시합니다. Python의 `sqlite3` 쿼리, `feedparser` 모듈 사용법 등 개발 전반의 코루틴을 작성하고 리뷰합니다.
*   **예상 산출물:** `main.py`, `crawler.py`, `processor.py`, `database.py` 등의 실제 구동 소스 코드.
*   **난이도: 최상 (Critical)**
    *   *이유:* 가장 많은 코드가 쏟아지고 예외 처리(Try-Except)가 정교하게 엮여야 하는 시기입니다. 긴 문맥(Long Context) 유지력과 코딩 정확도가 가장 우수한 모델(혹은 코딩 전화 모델)에 맡겨야 합니다.
*   **진행 상태:** 🔄 진행 대기

---

### 6단계 · 품질 관리 (Quality Assurance)
*   **작업 내용:** 작성된 코드가 로컬에서 메모리를 과도하게 점유하지 않는지, SQLite DB 파일이 덮어쓰기 권한에 문제가 없는지, 예외 상황(가상 인터넷 단절 테스트 등)에서 Fallback 로직이 잘 작동하는지 엣지케이스 테스트를 돌립니다.
*   **예상 산출물:** 단위 테스트(`pytest`) 코드 또는 로그(Log) 기반의 디버깅 리포트.
*   **난이도: 중 (Medium)**
    *   *이유:* 이미 작성된 코드를 기반으로 에러를 잡고 테스트 코드를 짜는 과정이므로, 논리성이 준수한 미들급 이상의 모델로도 커버 가능합니다.
*   **진행 상태:** 🔄 진행 대기

---

### 7단계 · 배포 및 운영 설계 (Deployment & Operations)
*   **작업 내용:** Mac 내장 스케줄러에 종속되지 않는 범용성(OS-Agnostic) 확보를 위해 파이썬 `schedule` 라이브러리를 활용한 백그라운드 스케줄러 스크립트(`scheduler.py`)를 개발합니다. 매일 정해진 시각에 텔레그램 메시지를 쏘고, GitHub에 `git push` 하는 자동화 파이프라인(CD)을 구동시킵니다.
*   **예상 산출물:** `scheduler.py` 실행 환경 및 `deploy.py` 배포 동기화 스크립트.
*   **난이도: 상 (High)**
    *   *이유:* OS 레벨의 쉘 프로그래밍 권한과 Git 워크플로우에 대한 명확한 이해, 텔레그램 API 네트워크 보안 등 DevOps 지식이 필요하므로 하이엔드 모델이 안전합니다.
*   **진행 상태:** 🔄 진행 대기

---

### 8단계 · 회고 및 진화 (Retrospective & Evolution)
*   **작업 내용:** 모든 시스템이 무사히 돌아가면, 프론트엔드의 부하(과거 JSON 정크 파일 등)를 분석하고, 추후 카테고리 확장을 대비해 리팩토링할 기술 부채(Technical Debt)를 문서화 해둡니다.
*   **예상 산출물:** 시스템 한계점 및 v2 고도화 대비 문서.
*   **난이도: 하 (Low)**
    *   *이유:* 프로젝트 전체를 훑어보는 가벼운 요약 및 아이디에이션 단계입니다.
*   **진행 상태:** 🔄 진행 대기
