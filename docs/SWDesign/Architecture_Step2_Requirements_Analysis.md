# 아키텍처 2단계: 요구사항 분석 (Requirements Analysis)

본 문서는 **[AI News Archive: 아키텍트의 8단계 실행 계획서]**의 2단계 산출물로, 기능적 요구사항(FR) 이면에 숨겨진 비기능적 요구사항(NFR)과 시스템 제약사항에 따른 품질 속성(Quality Attributes)을 식별하고 우선순위를 정의합니다.

---

## 1. 비기능 요구사항 (Non-Functional Requirements - NFR)

단순한 개인용 데일리 크롤러 봇처럼 보이지만, '제로 코스트(0원)' 제약 조건과 '로컬 LLM' 구동이라는 한계 상황 때문에 이하의 NFR들이 1순위로 지켜져야 합니다.

### 1-1. 성능 및 리소스 제한 (Performance & Resource Limits)
*   **LLM 추론 한계:** M4 Mac Mini의 통합 메모리(RAM)를 다른 로컬 작업과 공유하므로, Ollama 모델(기본 `gemma3:4b`, 추후 확장에 따라 `12B`급 모델까지 스왑 지원)이 한 번에 너무 많은 VRAM을 선점하지 않도록 파이프라인은 기사 요약을 '병렬(Parallel)'이 아닌 **'순차적(Sequential 배치)'**으로 실행해야 합니다.
*   **응답 제한 타임아웃(Timeout):**
    *   **크롤링:** 각 RSS Feed 접근 시 최대 **5초** 대기 (초과 시 과감히 스킵).
    *   **LLM 생성:** 각 기사당 요약 반환 대기 시간 최대 **30초** (초과 시 Fallback 로직 발생).

### 1-2. 스토리지 및 트래픽 (Storage & Bandwidth)
*   **백엔드 스토리지 (Local DB):** SQLite DB 블로트(Bloat) 현상을 막고 검색 속도를 보장하기 위해 DB 사이즈는 최소한으로 유지되며, 원문 기사 전체 내용을 저장하지 않고 '제목+요약문+해시'만 저장합니다.
*   **프론트엔드 트래픽 (GitHub Pages):** Github Pages는 트래픽 제한(초과 시 차단)이 존재하므로, 사용자가 웹사이트 접속 시 수집된 전체(All) JSON을 다운받게 해서는 안 됩니다.
    *   반드시 일자별(`daily/YYYY-MM-DD.json`), 주간별(`weekly/Week_XX.json`)로 **JSON 파일을 쪼개어(Sharding)** 필요한 날짜의 데이터(약 10~50KB)만 Fetch 하도록 제한합니다.
    *   정적 JSON 파일은 유지보수를 위해 최근 **30일치(1개월)** 만 보관하고 그 이전 파일은 자동 삭제(Retention)되게 구성합니다.

### 1-3. 가용성 및 내결함성 (Availability & Fault Tolerance)
*   일부 언론사 서버 점검, 일시적 인터넷 끊김, Ollama 프로세스 다운 등의 상황이 발생해도 파이프라인 전체가 멈추거나 패닉(Panic) 종료를 일으키면 안 됩니다.
*   "절반의 성공이라도 기록한다"는 철학 아래, 실패한 기사는 건너뛰고 정상 처리된 기사들만이라도 모아서 아침 7시에 반드시 배포(Publish)되어야 합니다.

---

## 2. 품질 속성 우선순위 (Quality Attributes Priority)

본 프로젝트의 아키텍처 설계 시 시스템 트레이드오프(Trade-off)가 발생할 경우 아래의 우선순위를 따릅니다.

1.  **비용 절감 (Cost Efficiency):** 무조건 0원. 클라우드 DB, AWS 람다 등 유료화 전환이 가능한 솔루션은 즉각 거부합니다.
2.  **안정성 및 자가 치유 (Resilience / Fault Tolerance):** 중간에 에러가 나도 스크립트가 죽지 않고 끝까지 돌아 배포를 마쳐야 합니다. 에러 알림은 로그 파일로만 남깁니다.
3.  **유지보수성 (Maintainability):** 복잡도가 높은 도커(Docker)나 쿠버네티스, n8n 등 제3자 의존성을 피하고, 오직 Python 기본 모듈 위주로 코드를 짜서 훗날 다른 개발자가 수정하기 쉽게 만듭니다.
4.  **성능 (Performance):** 로컬 스크립트 실행 속도가 3분 걸리든 5분 걸리든 상관없습니다. 아침 7시 이전에만 돌아서 결과물만 내놓으면 됩니다. (속도 최적화 < 메모리 절약)
