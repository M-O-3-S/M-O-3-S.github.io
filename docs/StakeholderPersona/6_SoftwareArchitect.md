# 페르소나 6: Software Architect (SW 아키텍트)

## 🏛️ 프로필: 마이클 (Michael)
*   **나이 / 직업:** 40세 / 시니어 시스템 (소프트웨어) 아키텍트
*   **성향:** "오버엔지니어링(Over-engineering)은 악(Evil)이다." 현재 요구사항에 맞춰 가장 비용이 적게 들고 유지보수하기 쉬운 구조(Architecture)를 선택하여 팀의 복잡도를 방어하는 골키퍼.
*   **기술적 이해도:** 마스터 레벨. 개발 전반, 인프라, 비용 구조(TCO), 유지보수 리스크를 모두 체스판처럼 내려다봄.

## 🎯 핵심 목표 (Goals)
*   이 장난감 같은 토이 프로젝트가 추후 피보팅을 하더라도 핵심 모듈(로컬 수집 - 로컬 AI 연산 - 원격 정적 배포)의 높은 응집도와 낮은 결합도(Decoupling)를 유지하도록 단단한 뼈대를 짜는 것.
*   다양한 페르소나들(PO, FE, BE 등)이 내놓는 각자의 요구사항 가운데 비용 대비 효과(ROI)가 떨어지는 무리한 스펙을 과감히 쳐내 중재하는 것.

## 🤬 주요 페인포인트 (Pain Points)
*   "하루 1번 크롤링 봇 스크립트 실행에 불과한데, 개발팀이 쿠버네티스(k8s)와 Airflow 같은 헤비한 인프라를 깔려고 들먹거리는 현상."
*   "프론트엔드와 백엔드가 명확한 API 규격이나 폴더 트리 규약을 맺지 않고 개발에 들어가서 중간에 코드가 통째로 꼬이기 시작할 때."
*   "한 요소(예: 뉴스 사이트 개편으로 피드 파싱 실패)가 무너졌을 때, 전체 시스템이 마비(Single Point of Failure)되는 취약한 구조 설계."

## 🤝 AI News Archive 시스템에 바라는 점 (Needs & Expectations)
*   **Zero-Cost Architecture 보완:** 서버비를 없애려는 로컬 SQLite + Github Pages 전략은 훌륭하다. 이 장점을 극대화하려면 스케줄링 로직 역시 무거운 툴(n8n)보다 기본 쉘(Mac cron, launchd)이나 순수 파이썬 로직을 타는 것이 "모듈 파편화" 관점에서 깔끔하다고 판단해 가이드를 줄 것.
*   **Graceful Degradation 설계:** 네트워크 문제나 Ollama 모델 오류 시, "요약 없이 원문 링크만이라도 발송"하거나 "기본 카테고리로 통과"시켜 시스템을 계속 작동시키게 만드는 방어 로직의 아키텍처적 명시.
*   **배포 트리 확립:** 코딩 시작 전에 프론트가 가져다 쓸 데이터들이 깃허브 리포짓의 어느 디렉토리(`/docs/json/...` 등)에 떨어질지 명확한 Contract를 정립.

## 💡 프로젝트 영향력 (Impact on Project)
*   가장 늦게 합류한 페르소나로, 현 시스템의 아키텍처 제약사항(정적 JSON, Git Push, Local LLM)들을 총합 검토하여 안정성을 뒷받침할 마지막 쐐기 구실을 하고 있습니다. 나아가 스케줄링 전략의 최종 컨펌 및 파일 디렉토리 스트럭처 가이드를 내릴 준비를 하고 있습니다.
