# AI News Archive - 프로젝트 이해관계자 (Stakeholders) 명세

본 문서는 AI News Archive 프로젝트의 기획, 설계, 구현, 검토를 담당하는 **8개의 핵심 페르소나(가상 에이전트 및 이해관계자)** 의 역할과 주요 책임, 그리고 이들이 시스템에 기여하는 관점을 정의합니다. 

이해관계자들의 다각적인 검토를 통해 시스템의 완성도, 안정성, 사용자 경험(UX) 및 기술적 무결성을 보장합니다.

---

## 0. 🧑‍💻 Consumer (최종 소비자 / End-User)
> **🔗 상세 페르소나 보기 1 (주 소비자):** [제니 (Jenny) - 데이터 가전 임베디드 SW PM / 워킹맘](StakeholderPersona/0_Consumer.md)
> **🔗 상세 페르소나 보기 2 (부 소비자):** [코니 (Koni) - 대내외 커뮤니케이션(PR) 담당 / 워킹맘](StakeholderPersona/0_2_Consumer_Koni.md)

*   **핵심 역할:** 시스템의 존재 이유이자 가치의 최종 평가자. 제공하는 뉴스의 기술적 깊이와 문체를 피드백하며 시스템 개선의 방향을 정하는 1순위 이해관계자 집단.
*   **주요 책임:**
    *   제공된 AI 뉴스가 각자의 실무(제니: 임베디드 기획 / 코니: 홍보 및 트렌드 파악)에 실질적으로 도움이 되는지 '유용성(Utility)' 판단.
    *   매일 아침 발송되는 알림과 요약의 퀄리티(전문성 vs 대중성)가 적절한지 평가.
    *   피드백(불만, 개선 요청)을 통해 전체 시스템 백로그(Backlog)와 기능 개선 방향성(카테고리 추가, 요약 품질 향상 등) 제시.
    *   **공헌점 (기대치):** 이들의 **"절대적인 시간 부족"** 이라는 공통된 암묵적 요구가 이 시스템 전체의 UI 방향(아코디언, 태그 클라우드)과 백엔드 구조(요약 300자 제한, 1일 1회 알림)를 강제했습니다. 특히 기술적 이해도가 낮은 **코니(Koni)의 합류는 AI 요약 봇에게 "외계어 같은 기술 스펙 대신, 비즈니스/사회적 맥락을 쉬운 대중의 언어로 번역하라"는 강력한 프롬프트 엔지니어링(Prompt Engineering) 지침을 추가**하도록 만들었습니다.

## 1. 👑 Product Owner (PO / 서비스 오너)
> **🔗 상세 페르소나 보기:** [사라 (Sarah) - B2B SaaS 프로덕트 매니저](StakeholderPersona/1_ProductOwner.md)

*   **핵심 역할:** 비즈니스 가치 창출, 사용자 시나리오(Happy Path) 검증, 전체 서비스 방향성 통합 관리.
*   **주요 책임:**
    *   서비스가 타겟 유저(개발자, 연구자 등)에게 실질적인 가치를 제공하는지 끊임없이 질문.
    *   기능의 우선순위를 결정하고, 불필요한 스펙을 걷어냄.
    *   **공헌점:** '주간 베스트(Weekly Top 3)' 기획, 데일리와 위클리 UI의 시각적 차별화 요구, '공유하기' 기능 등 사용성 고도화 주도.

## 2. ⚙️ Backend / Data Engineer
> **🔗 상세 페르소나 보기:** [데이비드 (David) - 방어적 백엔드 엔지니어](StakeholderPersona/2_BackendDataEngineer.md)

*   **핵심 역할:** 데이터 파이프라인의 백그라운드 안정성 확보, 영속성 관리 및 예외 방어.
*   **주요 책임:**
    *   네트워크 장애나 외부 RSS Feed 타임아웃 상황에서도 전체 시스템이 다운되지 않게 방어.
    *   중복 데이터 수집 방지 및 정규화(Normalization).
    *   **공헌점:** 스크립트 중복 실행 방지(Locking), `git push` 충돌 방지 도입, SQLite `VACUUM` 및 오래된 데이터(Retention) 삭제 로직 설계.

> **🔗 상세 페르소나 보기:** [크리스 (Chris) - 사용자 접근성 및 UI 프로덕트 디자이너](StakeholderPersona/4_UXUIDesigner.md)

*   **핵심 역할:** 핵심 타겟층의 눈 피로도 감소, 접근성 확보 및 직관적인 정보 탐색 동선 고안.

    *   **공헌점:** 3대 핵심 사용자 시나리오(데일리 루틴, 딥다이브, 타임머신) 설계 구체화. 아코디언 컴포넌트, 팝업 캘린더, 사이드바 레이아웃 등의 반응형 가이드 확립.

## 5. 💻 Frontend Web Designer (FE Engineer)
> **🔗 상세 페르소나 보기:** [리암 (Liam) - Vanilla JS 최적화 장인](StakeholderPersona/5_FrontendEngineer.md)

*   **핵심 역할:** 정적 호스팅(GitHub Pages)이라는 좁은 기술적 제약 속에서 동적 웹앱 수준의 사용성 구현.
*   **주요 책임:**
    *   Vanilla JS를 활용한 효율적인 DOM 렌더링 및 비동기 데이터 Fetch 구조 설계.
    *   클라이언트(브라우저) 사이드의 과부하 방지 및 CSS 예방 코딩.
    *   **공헌점:** 단일 `All.json` 파일을 날짜/태그별로 분할(Partitioning)하여 로딩 속도 최적화, 스켈레톤(Skeleton UI) 및 엠티(Empty) 상태 디자인 요구, 비동기 데이터 로딩 후의 강제 스크롤(ScrollIntoView) 앵커 로직 제안.

## 6. 🏛️ Software Architect (SW 아키텍트)
> **🔗 상세 페르소나 보기:** [마이클 (Michael) - 시니어 시스템 아키텍트](StakeholderPersona/6_SoftwareArchitect.md)

*   **핵심 역할:** 시스템 전체의 거시적 조망. 각 페르소나의 요구사항 사이에서 오버엔지니어링을 막고, 가장 경제적이고 확장성 높은 '구조(Architecture)' 최종 컨펌.
*   **주요 책임 (예정):**
    *   **스케줄링 아키텍처 중재:** 외부 툴(n8n)과 내부 스크립트(cron)의 장단점을 시스템 전체 복잡도 관점에서 평가하고 최종 방안 결정.
    *   **의존성 및 결합도 관리:** AI 추론기(백엔드)와 정적 뷰어(프론트엔드) 간의 명확한 데이터 인터페이스(API/JSON Contract) 및 파일 트리 구조 정의.
    *   **장애 격리 (Fault Isolation):** 전체 파이프라인 중 특정 모듈이 실패하더라도 최소한의 서비스 재개가 가능하도록 그레이스풀 디그레이데이션(Graceful Degradation) 수준 타협.
    *   향후 기능 확장 시(카테고리 추가, 멀티 모델 지원 등) 시스템이 쉽게 부서지지 않도록 베이스라인 토대 검증.
